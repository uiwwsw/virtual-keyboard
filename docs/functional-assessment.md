# 기능 테스트 및 개선 제안

## 테스트 요약
- `npm run build`로 TypeScript 빌드와 프로덕션 번들을 생성하여 주요 빌드 파이프라인을 검증했습니다.
- `npm run lint`로 ESLint 정적 분석을 실행해 린트 규칙 준수 여부를 확인했습니다.

## 추가 수동 테스트 권장 사항
- 데스크톱과 모바일 브라우저에서 `npm run dev` 후 노출되는 샘플 페이지를 열어 가상 키보드 노출 조건, 한글/영문 전환, 선택 영역 유지 등의 동작을 직접 확인해 주세요.
- 모바일 Safari/Chrome에서 입력란 포커스 시 뷰포트 변화(키패드 위치 및 body padding 보정)가 자연스럽게 작동하는지 검증이 필요합니다.

## 개선 제안
### 1. 접근성 있는 입력 역할 제공
현재 `VirtualInput`은 `<div role="textbox">` 구조로 구현돼 있어 스크린 리더가 기본 입력 요소처럼 인식하지 못하며, 글자 클릭을 `role="button"`으로 감싸 포커스 이동이 어려울 수 있습니다.【F:src/components/Input.tsx†L368-L453】 표준 `<input>` 혹은 `contentEditable` 요소 사용과 ARIA 속성 정교화를 통해 키보드 사용자와 보조 공학 호환성을 높일 수 있습니다.

### 2. 사용자 에이전트 기반 모바일 판별 개선
모바일 환경 여부를 `navigator.userAgent` 정규식으로만 판단하고 있어 서버 사이드 렌더링 시 `navigator` 미존재로 에러가 발생할 수 있습니다.【F:src/utils/isMobileAgent.ts†L1-L7】 안전한 접근을 위해 브라우저 존재 여부를 확인하거나 뷰포트 정보, 포인터 타입 등 다른 신호를 조합하는 방식을 권장합니다.

### 3. 커스텀 키 레이아웃 이벤트 매핑 보완
가상 키패드는 모든 키에 `KeyboardEvent`의 `code`를 `Key${value.toUpperCase()}` 형태로 고정해 전달합니다.【F:src/components/Keypad.tsx†L72-L200】 그러나 `010`처럼 여러 글자로 된 사용자 정의 키에서는 실제 물리 키 코드와 맞지 않아 입력 처리에 실패할 수 있습니다. `layout` 항목에 별도 `code`를 허용하거나 `value` 길이에 따라 적절한 코드/키 값을 분리해 전달하도록 보완하는 것이 좋습니다.

### 4. 시프트 키 동작 일관성
시프트 키를 한 번 누르면 토글 상태가 유지돼 연속 입력 시 대문자/기호가 계속 입력됩니다.【F:src/components/Keypad.tsx†L72-L200】 실제 모바일 키보드처럼 첫 입력 후 자동으로 해제되거나, 시프트 유지 여부를 UI로 명확히 표시하면 사용자 혼란을 줄일 수 있습니다.

### 5. 키패드 포커스와 포인터 이벤트 처리
키패드 컨테이너는 `tabIndex={-1}`로 지정돼 키보드 포커스를 받지 못해 보조공학 사용자가 제어하기 어렵습니다.【F:src/components/Keypad.tsx†L33-L200】 포커스 가능한 요소로 전환하고 키보드 입력을 지원하면 접근성이 향상됩니다. 또한 외부 클릭 시 닫힘 애니메이션이나 전환 효과를 추가하면 모바일 UX가 자연스러워집니다.
